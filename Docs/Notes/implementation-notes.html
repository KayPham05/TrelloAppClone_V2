2026-06-13 00:00 Asia/Bangkok - Member invite autocomplete implementation

Design decisions

  • Backend board autocomplete validates that boardId belongs to the supplied workspaceId before returning suggestions. This is stricter than the plan sample and prevents cross-workspace suggestion leakage if the client sends mismatched IDs.
  • Flutter tests and imports use the real package name apptreolon from pubspec.yaml, not the plan placeholder trellon_flutter.
  • MemberInvitePicker only calls Navigator.pop after a successful submit when the current navigator can pop. This keeps the widget usable both inside modal sheets and inside widget tests/embedded contexts.
  • Board invite autocomplete now receives workspaceId through BoardDetailPage -> BoardMenuSheet -> BoardMembersManageSheet, because board suggestions must be constrained to workspace members.
  • Updated BoardDetailPage from deprecated onReorder to onReorderItem while touching the file, because the current Flutter analyzer treats the old callback as an issue. The destination index no longer needs manual adjustment.

Deviations

  • Không dùng trực tiếp UserLocalDataSource() như plan snippet; các UI mới lấy qua serviceLocator<UserLocalDataSource>() để khớp dependency injection hiện có.
  • Chưa xóa các helper method legacy không còn được gọi trong một số page vì file đang có chuỗi mojibake cũ gây khó patch an toàn; active entry points đã chuyển sang picker mới và file add_member_dialog.dart đã bị xóa.

Tradeoffs

  • Batch invite hiện gửi từng request tuần tự và reload một lần sau khi có ít nhất một invite thành công. Cách này đơn giản và giữ nguyên backend endpoint hiện tại, nhưng chưa tối ưu số request như một endpoint batch thật sự.
  • Board sheet nhận workspaceId nullable để không phá các luồng personal/loading cũ. Nếu workspaceId rỗng, backend autocomplete board sẽ trả danh sách rỗng thay vì gợi ý sai scope.

Open questions

  • Có muốn thêm endpoint batch invite thật sự ở backend để giảm số request khi chọn nhiều người không?
  • Có muốn xóa triệt để các helper legacy trong page bằng một pass riêng xử lý encoding/mojibake không?

2026-06-13 12:30 Asia/Bangkok - Board rename and per-user starred boards

Design decisions

  • Board star state is persisted on the backend per user with a new UserStarredBoard table and StarredBoard API, instead of storing it locally in Flutter.
  • Recent-board DTOs, starred-board DTOs, all-board DTOs, and workspace DTO board lists now expose isStarred so Flutter can render the same state in home, board list, and workspace screens.
  • Workspace board rename reuses the existing PUT /boards/{id} update path and shows a snackbar after success.
  • Starred boards are displayed before recent boards, and both sections paginate on the client with a maximum of 5 boards per page.
  • Workspace board lists sort starred boards to the top while preserving the previous relative order inside starred and non-starred groups.
  • C#/TodoAppAPI/.gitignore now keeps migrations ignored by default but explicitly unignores 20260613123000_AddUserStarredBoard.cs, because the new per-user star table needs a migration in source control.
  • Added 20260613123000_AddUserStarredBoard.Designer.cs after EF CLI reported no pending migrations; the designer file supplies the [Migration] attribute EF uses to discover the migration.

Deviations

  • Pagination is currently client-side because the requirement was about visible section size, and the backend recent/starred lists are small bounded dashboard collections.
  • The board detail menu star icon remains a static menu affordance in this pass because the menu does not currently receive isStarred state; active star toggling is implemented from workspace board cards and reflected in home/list sections.

Tradeoffs

  • Using the existing board update endpoint avoids adding a rename-only endpoint, but required hardening UpdateBoardAsync so a rename request does not clear an existing background URL.
  • Client-side pagination keeps the API simple and matches the current dashboard usage, but backend pagination may be preferable later if users can accumulate many starred/recent boards.

Open questions

  • Do you want the static star button inside the board detail menu to become an active toggle too, with current star state passed into that sheet?
  • Should recent/starred pagination controls use text page counters as implemented, or a different visual treatment?
