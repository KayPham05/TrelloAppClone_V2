2026-06-13 23:55 Asia/Bangkok - Comment feature implementation checkpoint

Design decisions

  • Comment notifications now create Mention notifications for tagged board/card members and Comment notifications for assigned card members, excluding the commenter.
  • Users can tag by @username, email prefix, or full @email, including plus-address emails such as @member+qa@example.com.
  • Flutter uses the board member list for mention suggestions when available, falling back to assigned card members if the board member list has not been loaded.
  • Comment attachments are uploaded after the comment is created, then attached to the same comment in local state.
  • Edit/delete endpoints only update or remove the comment and broadcast the card detail realtime event; they do not call notification creation.

Deviations

  • Attachment UI currently renders compact file chips instead of a rich preview gallery. This keeps the first implementation generic for all file types and avoids duplicating the card attachment image editor flow.
  • Inbox-card comment attachments/edit/delete are disabled from the card detail UI because the implemented backend endpoints target board-card comments. Existing inbox comment creation remains unchanged.
  • A pre-existing ignored migration named 20260613152714_AddCommentAttachmentSupport was left untouched because it appears to create a broad schema baseline. The source-controlled migration for this feature is 20260613154422_AddCommentAttachment.

Tradeoffs

  • Uploading comment files after comment creation avoids adding a mixed multipart create-comment endpoint, but a failed file upload can leave the text comment created without some attachments.
  • Using a separate CommentAttachment table keeps comment files isolated from card-level attachments, but adds parallel DTO/model mapping on both backend and Flutter.

Open questions

  • Do you want edit-comment mode to reuse the full mention suggestion bottom sheet, or is the current inline edit enough?
  • Should comment attachment chips open the uploaded URL in-app/browser when tapped?
  • Should inbox comments support the same edit/delete/attachment behavior in a later backend pass?
