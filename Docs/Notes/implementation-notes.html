Gemini Project Analysis Implementation Notes

Ghi chú triển khai cho tính năng Gemini API phân tích và báo cáo tiến độ dự án.

2026-06-02 00:00 Asia/Bangkok - Phase 1 kickoff

Design decisions

  • Chọn REST typed HttpClient cho Gemini thay vì thêm package Google.GenAI trong Phase 1. Lý do: REST docs hiện mô tả structured JSON schema rõ hơn, không cần tải dependency mới, và vẫn giữ API key ở backend.
  • Không tạo DB phase trong lượt MVP này. Plan mới nói DB report history là later phase; MVP dùng IMemoryCache 5 phút.
  • Flutter Phase 2 chỉ expose Board analysis. Backend Phase 1 vẫn tạo service/route cho Workspace, Board, Card theo plan.

Deviations

  • Nếu plan còn nhắc Google.GenAI như lựa chọn ưu tiên, implementation thực tế dùng REST để giảm rủi ro dependency và giữ structured output kiểm soát được.

Tradeoffs

  • REST cần tự parse response candidates[].content.parts[].text, nhưng đổi lại dễ mock test và không phụ thuộc SDK API surface.

Open questions

  • DB report history có cần triển khai ngay sau MVP hay để sprint riêng?
2026-06-02 00:00 Asia/Bangkok - Phase 1 backend implementation

Design decisions

  • Thêm AnalysisResult wrapper để service phân biệt Success, NotFound, và Forbidden. Controller chuyển wrapper này thành HTTP status tương ứng.
  • Backend analysis authorization dùng role chặt hơn view thường: Board Owner/Admin/Editor được phân tích; Board Viewer bị từ chối. Workspace Owner/Admin được phân tích workspace hoặc board thuộc workspace.
  • Gemini lỗi, thiếu key, timeout, hoặc JSON không hợp lệ đều fallback về DTO metric-only để frontend không crash.

Deviations

  • Không thêm NuGet Google.GenAI; REST client được dùng cho Phase 1.
  • Không tạo DB migration/report history vì plan đã chuyển phần này sang later phase.

Tradeoffs

  • Metric hoàn thành card dùng rule explicit: status/list name chứa done/completed/hoàn thành/xong. Đây là heuristic vì repo chưa có trạng thái workflow chuẩn.
  • Cache hiện mutate DTO cached bằng cách set Cached = true khi hit cache. Cách này đơn giản cho MVP, nhưng nếu sau này cần immutable DTO thì nên clone report trước khi trả.

Open questions

  • Có cần thay query param userUId bằng user id lấy từ JWT claim ở một phase hardening riêng không?
2026-06-02 00:00 Asia/Bangkok - Phase 2 frontend implementation

Design decisions

  • Frontend dùng feature folder features/ai_analysis với các lớp data/domain/presentation riêng để bám theo kiến trúc Flutter hiện có.
  • Màn hình Phase 2 mở từ Board Detail và gọi endpoint analysis/board/{boardUId}?userUId=...; repository/usecase vẫn hỗ trợ Workspace/Card để không lệch contract backend Phase 1.
  • Nút Báo cáo AI chỉ hiển thị cho Board Owner/Admin/Editor. Viewer không thấy entry point để khớp authorization backend.
  • Parser model chấp nhận cả schema DTO Phase 1 (detail, doneCards, label) và schema mô tả trong test/plan (description, completedCards, name).

Deviations

  • Chưa thêm route global trong router vì board detail hiện đang dùng Navigator.push/MaterialPageRoute cho nhiều luồng cục bộ; entry point AI dùng cùng kiểu này để scope thay đổi nhỏ.
  • Chưa expose Workspace/Card analysis trong UI mặc dù data layer đã hỗ trợ, vì plan Phase 2 ưu tiên Board analysis trong board detail.

Tradeoffs

  • Không thêm package chart/progress mới; dùng widget Flutter mặc định để tránh dependency và giữ MVP nhẹ. Khi cần dashboard giàu hơn có thể bổ sung chart library ở phase sau.
  • UI hiển thị fallback banner khi rủi ro và đề xuất rỗng. Cách này đơn giản hơn việc backend phải thêm field fallback riêng, nhưng có thể nhận nhầm board thật sự không có risk/suggestion.

Open questions

  • Có muốn thêm entry point phân tích cho Workspace menu và Card detail ngay ở phase sau không?
  • Có cần lưu lịch sử báo cáo phía client/server để so sánh tiến độ theo thời gian không?

2026-06-04 Asia/Bangkok - AI report metrics, charts, and force refresh improvement

Design decisions

  • Không thay đổi database. Các số liệu mới được tính từ dữ liệu hiện có: Cards.Status, Cards.DueDate, checklist/todo items, lists và board.
  • Backend chuyển tiến độ tổng thể sang logic dựa trên tỷ lệ thẻ hoàn tất theo Card.Status; checklist vẫn là chỉ số phụ, không kéo lệch phần trăm header.
  • Thêm forceRefresh vào endpoint phân tích để nút refresh và pull-to-refresh bỏ cache, đọc lại dữ liệu board mới nhất rồi ghi cache mới.
  • Backend luôn tạo summary, risks và suggestions deterministic từ metric trước; Gemini chỉ nâng chất lượng diễn giải khi trả JSON hợp lệ.
  • Frontend thêm donut chart và bar chart bằng CustomPainter/LinearProgressIndicator thay vì thêm chart package mới.

Deviations

  • Không triển khai mini timeline due-date trong lượt này để tránh mở rộng DTO danh sách card chi tiết. Thay vào đó thêm metric dueSoonCards và risk/suggestion tương ứng.
  • Không thêm report history/cloud storage trong lượt này vì yêu cầu hiện tại tập trung vào chất lượng báo cáo hiện tại và refresh dữ liệu mới.

Tradeoffs

  • Custom chart nhẹ giúp tránh dependency mới, nhưng khả năng tương tác chart còn giới hạn so với chart library chuyên dụng.
  • Status distribution dùng các nhóm ổn định todo, inProgress, completed, other; các status lạ được gom vào other để không làm sai tiến độ.
