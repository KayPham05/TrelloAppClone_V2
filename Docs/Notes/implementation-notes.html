Design decisions
Notification creation from business services is now best-effort after the business save succeeds. Domain services call TryCreateInternalAsync / TryCreateManyInternalAsync so a notification write/send failure is logged but does not roll back the actual board, card, comment, member, or workspace operation.

No outbox, retry queue, or background notification dispatcher was added in this pass. This intentionally differs from a guaranteed-delivery design because the reviewed high-risk bug was false failure of successful business actions, not durable notification retry.
Due-date scheduled reminders remain out of scope. The current implementation covers due-date set/change notifications only; timed reminders still require a future scheduler specification.
Navigation does not use arbitrary backend deep-link strings as the source of truth. This is a deliberate safety and correctness deviation: routes are built from validated typed IDs so stale or malformed links do not navigate blindly.
The manual realtime smoke test with two signed-in devices and a real backend was not executed in this documentation pass. Automated tests cover the pure Flutter/backend logic, but live SignalR transport still needs manual confirmation.
Tradeoffs
Best-effort notification dispatch is simpler and safer for current domain operations than introducing an outbox schema now. The tradeoff is that notification delivery can still be lost if the notification write fails after the business transaction succeeds.
The navigation service resolves cards through existing board/card data instead of adding a new dedicated backend deep-link resolver. This avoids API expansion, but deleted or inaccessible cards are handled as non-navigable results.
Flutter tests focus on cubit/service behavior rather than a full widget/device flow. This keeps required tests deterministic in CI-style environments, but it does not replace final manual smoke testing for SignalR and real screen navigation.

Design decisions
Scheduled due-date reminder is now a required notification feature before the notification feature can be considered merge-ready. The scheduler rule is fixed: run every 15 minutes and notify assignees at exactly three milestones per card: 1 day before due date, 1 hour before due date, and when due date is reached.
Each reminder milestone must be sent at most once per card. Cards that are completed or have no assignees are skipped.
If a card due date is changed or removed, the reminder send history for that card must be reset so all milestones can be sent again from the new due-date state.
Typed notification context IDs are accepted as sufficient for navigation. No backend-generated canonical deep link is required for this phase.
The manual realtime/navigation smoke test is now documented as a required Definition of Done in Docs/Tests/notification-smoke-test.md.
Deviations
The previous due-date reminder deviation is no longer accepted as final behavior. The current code still lacks the scheduled reminder worker/history reset behavior, so this is now a pending implementation gap before merge rather than an approved out-of-scope item.
No deviation is needed for backend deep links because typed IDs are the confirmed design.
Tradeoffs
The scheduler requirement keeps reminder delivery inside the backend instead of pushing this responsibility to Flutter clients. This is more reliable for offline users, but it requires backend state to track which milestone notifications were already sent.
The smoke test remains manual because it verifies live SignalR behavior across two active sessions, which the current automated test setup does not cover end-to-end.
Open questions
None for navigation: typed IDs are enough.
None for the smoke-test merge gate: if all 6 scenarios pass, merge is allowed; if any scenario fails, open a bug ticket and do not merge.
Implementation question for the next code pass: choose the backend persistence shape for reminder send history, likely a small table keyed by card ID and reminder milestone so due-date changes can reset the card's reminder records safely.

2026-05-26 21:47:52 +07:00

Design decisions
Scheduled due-date reminder is now implemented in the backend. CardDueDateReminderHostedService runs once when the API starts and then repeats every 15 minutes with server-local DateTime.Now.
Reminder send history is persisted in CardDueDateReminderDeliveries with a unique key on (CardUId, Milestone). The scheduler records the milestone marker before creating notifications so the database unique index can prevent duplicate delivery under concurrent scheduler runs.
The three reminder windows are implemented as fixed milestones: OneDayBefore for due dates within 24 hours but more than 1 hour away, OneHourBefore for due dates within 1 hour but still in the future, and DueNow for due or overdue cards.
Cards are skipped when they are deleted, completed, or have no assignees. Completed status matches the app's existing status strings: Completed, hoan_thanh, and hoÃ n thÃ nh.
Changing or removing a card due date resets all reminder history for that card through both UpdateDueDateAsync and the broader UpdateCard path.
NotificationService.CreateManyInternalAsync now keeps saved notification rows even if SignalR broadcast fails. This prevents the scheduler from treating a saved notification as failed only because realtime delivery failed.
Deviations
No Flutter code was changed for this pass because NotificationTypeEnum.dueDateReminder and typed card navigation already exist.
The scheduler does not send catch-up reminders for earlier milestones once the card is already in a later window. For example, an overdue card sends only DueNow, not backfilled 1-day or 1-hour reminders.
Tradeoffs
Using server-local time matches the existing backend date handling and the chosen plan, but it means due-date reminder behavior follows the API server timezone instead of each user's client timezone.
Delivery history is one row per card and milestone, not per assignee. This keeps the "once per card per milestone" rule simple, but if assignees are added after a milestone was sent, they will not receive that already-sent milestone until the due date changes and resets history.
The scheduler logs failures with Console.WriteLine to match the current service style. A production hardening pass should replace this with structured ILogger.
Open questions
Manual smoke test is still required before merge according to Docs/Tests/notification-smoke-test.md.
If product wants reminders to be per assignee rather than per card, the persistence key must change from (CardUId, Milestone) to include RecipientId.
